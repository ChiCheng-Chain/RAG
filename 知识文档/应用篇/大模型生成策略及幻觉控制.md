# 大模型生成策略与幻觉控制

前几篇我们搞定了 RAG 的数据准备和检索环节：原始文档经过分块、元数据管理、向量化，存入向量数据库；用户提问时，经过问题改写、意图识别、多通道检索、混合检索和重排序，拿到了最相关的知识片段。

但这只是完成了 RAG 的一半。另一半是生成——把这些知识片段交给大模型，让它生成最终回答。

这一篇，我们就来解决生成环节的核心问题：**怎么让大模型生成高质量的回答，尤其是怎么控制幻觉**。

## 什么是幻觉

幻觉（Hallucination）是大模型的老问题了。所谓幻觉，就是模型输出了**看似合理但实际错误**的内容。这些内容可能是：

- **事实性错误**：把“7天无理由退货”说成“15天”
- **无中生有**：编造一个不存在的政策、法规、数据
- **过度推断**：超出参考资料的范围，做出资料无法支持的结论

在 RAG 场景里，幻觉尤其棘手。因为用户使用 RAG 的目的就是**要准确**。如果模型无视检索到的知识，自顾自地发挥，那 RAG 就失去了意义。

## 幻觉从哪来

理解幻觉的来源，才能对症下药。RAG 系统中的幻觉主要有三个来源：

**第一，检索质量不高的锅**。如果检索到的内容本身就分散、不准确、或者相关性很低，模型巧妇难为无米之炊，只能靠预训练知识“补全”，从而产生幻觉。这是源头问题，后面第四章会重点讲。

**第二，Prompt 引导不当的锅**。Prompt 没有明确告诉模型“只能使用参考资料”“不要编造”，模型就会放飞自我。这是Prompt设计问题，第三章会讲。

**第三，模型本身的锅**。某些模型本身就喜欢“发挥”，尤其是在不确定的时候。这是模型选择和生成策略的问题，第二章会讲。

这篇文档会按照**从源头到末端**的顺序，依次讲解：

1. 生成策略基础：控制模型输出的核心参数
2. Prompt 约束：从指令层面限制模型“不敢编”
3. 检索增强：从源头减少幻觉产生的机会
4. 引用验证：检查输出是否“有据可依”
5. 多答案一致性：利用冗余检测错误
6. 事后校验：生成后的兜底防线

## 大模型生成策略基础

在讲幻觉控制之前，先搞清楚大模型生成的控制参数。这些参数直接决定了模型输出的风格和质量。

### 核心参数解析

调用大模型时，有几个关键参数会直接影响输出：

| 参数 | 作用 | 取值范围 | RAG 场景推荐值 |
|------|------|----------|----------------|
| temperature | 控制随机性 | 0~2 | 0~0.3 |
| top_p | 核采样阈值 | 0~1 | 0.7~0.9 |
| max_tokens | 最大输出 token 数 | 正整数 | 512~2048 |
| top_k | 候选词数量限制 | 正整数 | 一般不设 |

**temperature（温度）**：这个参数控制模型输出的确定性。

- 设为 0：模型总是选择概率最高的词，输出最确定，几乎每次相同输入的输出都一样
- 设为 1（默认值）：正常的随机性，输出有变化但保持合理性
- 设为更高：输出越来越随机、越有创意，但也越容易“胡说八道”

在 RAG 场景下，我们追求的是**准确**，不是创意。所以 temperature 通常设得很低，0 到 0.3 之间。有些系统直接设为 0。

**top_p（核采样）**：另一种控制随机性的方式。它不直接控制概率阈值，而是从概率最高的词开始累加，累计达到 top_p 之后就只从这个集合里选词。

通常 temperature 和 top_p 二选一就行，一起用效果反而可能变差。本系列统一用 temperature，更直观。

**max_tokens**：限制模型最多生成多少个 token。如果回答被截断，响应中会有 `finish_reason: length` 提示。设置这个值要预估回答长度——回答太短可能没说明白，回答太长浪费 token。

**top_k**：限制每一步从概率最高的 k 个词中选择。设得太小（如 1）等同于 greedy 搜索，很死板；设得太大（如 100）又太随机。这个参数通常不设置，用默认的 top_p 就够了。

### 生成参数对幻觉的影响

为什么低 temperature 能减少幻觉？因为幻觉往往发生在模型“自由发挥”的时候。高随机性意味着模型更可能选那些**看起来合理但实际错误**的词。

但也不是越低越好。极端情况下（temperature = 0），模型变得过于保守，可能只会机械地复述检索到的内容，缺乏必要的整理和衔接。

一个实用的建议：先用 temperature = 0 测试，看回答是否过于机械；然后调到 0.1~0.3，找到“准确”和“自然”的平衡点。

### Java 代码示例：构建生成请求

这段代码演示如何用 Java 构建一个大模型调用请求，包含生成参数配置。代码使用 OkHttp + Gson，和调用 Embedding API 的方式类似。

> **这段代码做什么？**
>
> 这是构建 RAG 场景下 Chat API 请求的完整过程：
> 1. **system 消息**：告诉模型是知识库问答助手，只能用参考资料回答
> 2. **user 消息**：把用户问题和检索到的知识片段组装在一起
> 3. **temperature = 0**：追求准确性，抑制随机性
> 4. **max_tokens = 1024**：预留足够的输出空间

```java
public class RAGChatRequestBuilder {

    // SiliconFlow API 地址
    private static final String API_URL = "https://api.siliconflow.cn/v1/chat/completions";
    private static final String API_KEY = "YOUR_API_KEY";
    private static final String MODEL = "Qwen/Qwen3-32B";

    // RAG 生成参数配置
    private static final double TEMPERATURE = 0.0D;  // 追求准确，抑制幻觉
    private static final double TOP_P = 0.3D;       // 与 temperature 配合使用
    private static final int MAX_TOKENS = 1024;     // 最大输出 token 数

    /**
     * 构建 RAG 场景下的 Chat 请求
     *
     * @param userQuestion 用户问题
     * @param contextDocs  检索到的知识片段（带编号和来源）
     * @return 完整的 Chat 请求 JSON
     */
    public static JsonObject buildRAGRequest(String userQuestion, 
                                              List<RetrievedChunk> contextDocs) {
        JsonObject requestBody = new JsonObject();
        
        // 1. 模型选择
        requestBody.addProperty("model", MODEL);
        
        // 2. 生成参数配置 - 抑制幻觉
        requestBody.addProperty("temperature", TEMPERATURE);
        requestBody.addProperty("top_p", TOP_P);
        requestBody.addProperty("max_tokens", MAX_TOKENS);
        requestBody.addProperty("stream", false);  // 非流式，便于调试

        // 3. 构建 messages 数组
        JsonArray messages = new JsonArray();

        // system 消息：定义角色和行为规则
        JsonObject systemMsg = new JsonObject();
        systemMsg.addProperty("role", "system");
        systemMsg.addProperty("content", buildSystemPrompt());
        messages.add(systemMsg);

        // user 消息：包含参考资料和用户问题
        JsonObject userMsg = new JsonObject();
        userMsg.addProperty("role", "user");
        userMsg.addProperty("content", buildUserPrompt(userQuestion, contextDocs));
        messages.add(userMsg);

        requestBody.add("messages", messages);
        
        return requestBody;
    }

    /**
     * 构建 System Prompt - 核心是约束模型只能使用参考资料
     */
    private static String buildSystemPrompt() {
        return """
            你是一个专业的知识库问答助手。
            
            # 行为准则
            1. 只能根据【参考资料】中的信息回答用户问题
            2. 不要使用你的预训练知识进行补充或推断
            3. 如果资料中没有相关信息，请明确告知用户
            4. 每个关键事实后必须标注来源编号
            """;
    }

    /**
     * 构建 User Prompt - 将检索结果格式化后发送给模型
     */
    private static String buildUserPrompt(String question, 
                                          List<RetrievedChunk> docs) {
        StringBuilder sb = new StringBuilder();
        
        // 格式化参考资料
        sb.append("【参考资料】\n");
        for (int i = 0; i < docs.size(); i++) {
            RetrievedChunk doc = docs.get(i);
            sb.append(String.format("[%d] 来源：%s\n", i + 1, doc.getSource()));
            sb.append(String.format("内容：%s\n\n", doc.getContent()));
        }
        
        // 用户问题
        sb.append("【用户问题】\n");
        sb.append(question);
        
        return sb.toString();
    }
}
```

这段代码展示了 RAG 场景下生成请求的构建方式。关键是 system prompt 中的约束——明确告诉模型“只能使用参考资料，不要用预训练知识”。这是幻觉控制的第一道防线。

## Prompt 层面的幻觉约束

Prompt 是你和模型沟通的唯一桥梁。Prompt 写得好不好，直接决定了模型的表现。

### Prompt 设计的核心原则

在 RAG 场景下，Prompt 设计的核心原则是**明确边界**：告诉模型什么能用、什么不能用、什么不能说。

具体来说，一个防幻觉的 Prompt 应该包含以下要素：

**第一，角色定义**。明确告诉模型它是“谁”，边界是什么。

```
你是一个专业的知识库问答助手，只能根据提供的参考资料回答问题。
```

**第二，知识边界**。明确告诉模型只能用什么、不能用什么。

```
只能使用【参考资料】中的信息进行回答。不要使用你的预训练知识。
如果参考资料中没有相关信息，请明确告知用户“我没有找到相关信息”。
```

**第三，引用规则**。要求模型标注信息来源。

```
每个关键事实后必须紧跟引用编号，例如：……[1]
没有引用的事实不要输出。
```

**第四，诚实兜底**。告诉模型不确定时怎么说。

```
如果资料中没有给出明确答案，请直接说“我不确定”，不要编造。
```

### 典型 Prompt 模板

基于上面的原则，这里提供一个 RAG 场景下可直接使用的 Prompt 模板。模板分两部分：系统指令和用户输入。

> **这个 Prompt 模板的设计思路**
>
> 1. **分层约束**：先给角色（我是谁），再给规则（能做什么、不能做什么），最后给格式要求
> 2. **明确边界**：“只能根据”“不要使用”这种祈使句，让模型没有歧义
> 3. **引用强制**：每句事实后面必须跟引用，没有引用就不输出
> 4. **诚实兜底**：主动告诉模型不确定时怎么说，避免它硬凑答案

```java
/**
 * RAG 场景下的防幻觉 Prompt 模板
 */
public class AntiHallucinationPromptTemplate {

    /**
     * 构建完整的 Prompt（包含 system 和 user 部分）
     */
    public static String build(String userQuestion, List<RetrievedChunk> chunks) {
        return buildSystemPrompt() + "\n\n" + buildUserPrompt(userQuestion, chunks);
    }

    /**
     * System Prompt - 核心防幻觉指令
     */
    private static String buildSystemPrompt() {
        return """
            # 角色定义
            你是一个专业的知识库问答助手。

            # 知识边界（必须严格遵守）
            1. 你只能根据【参考资料】中的信息回答用户问题
            2. 禁止使用你的预训练知识进行补充、推断或联想
            3. 如果【参考资料】中没有相关信息，明确告知用户

            # 引用规则（必须严格遵守）
            1. 每个关键事实后必须紧跟引用编号，格式：[数字]
            2. 没有引用编号的事实，不要输出
            3. 引用编号必须对应【参考资料】中的编号

            # 回答风格
            1. 回答要简洁、直接，不要绕弯子
            2. 只输出事实，不要输出推理过程
            3. 如果资料足以回答，直接给出答案
            """;
    }

    /**
     * User Prompt - 包含参考资料和问题
     */
    private static String buildUserPrompt(String question, List<RetrievedChunk> chunks) {
        StringBuilder sb = new StringBuilder();
        
        sb.append("【参考资料】\n");
        sb.append("以下是可供你参考的知识片段：\n\n");
        
        for (int i = 0; i < chunks.size(); i++) {
            RetrievedChunk chunk = chunks.get(i);
            sb.append(String.format("[%d]\n", i + 1));
            sb.append(String.format("来源：%s\n", chunk.getMetadata("title")));
            sb.append(String.format("内容：%s\n\n", chunk.getContent()));
        }
        
        sb.append("【用户问题】\n");
        sb.append(question);
        
        return sb.toString();
    }
}
```

### 结构化输出的辅助约束

有时候我们需要模型输出结构化的内容（比如 JSON），这时候要在 Prompt 里明确格式要求。

```json
# 输出格式要求
请按以下 JSON 格式输出答案：
{
    "answer": "直接回答用户问题",
    "references": ["[1]", "[2]"],  // 使用的引用编号列表
    "confidence": "high/medium/low",  // 答案的置信度
    "need_more_info": false  // 是否需要更多信息
}
```

结构化输出的好处是：便于程序解析、便于评估质量、便于统计引用覆盖率。

### Prompt 约束的效果与局限

Prompt 约束是幻觉控制的第一道防线，优点是**零成本、见效快**——只需要改 Prompt，不需要改架构。

但它也有局限：

- **模型不一定完全遵守**：越弱的模型，遵守指令的能力越差
- **无法检测事实错误**：模型可能“自信地编造”，引用也可能“空挂”（引用了但内容不支持）
- **过度约束导致机械**：temperature = 0 时，模型可能只是机械复述，缺乏整理

所以 Prompt 约束需要配合其他手段（检索增强、引用验证、事后校验）一起使用。

## 检索增强——从源头减少幻觉

好的检索是幻觉控制的第一道防线。如果检索到的内容本身就不准确、不完整，模型再有本事也白搭。

这一节结合项目文档，讲解如何通过优化检索质量来减少幻觉。

### 检索质量对幻觉的影响

为什么检索质量和幻觉相关？因为 RAG 的本质是**让模型基于知识回答**。如果知识不对、知识不全，模型要么“巧妇难为无米之炊”，要么只能靠预训练知识“补全”——这一补全，就容易产生幻觉。

具体来说，检索质量影响幻觉的两种典型情况：

**第一种，检索到的内容相关性低**。用户问 A 问题，检索回来的却是 B 知识。模型不知道这是不相关的，只能硬着头皮用 B 来回答 A，结果答非所问。

**第二种，检索到的内容分散**。用户问一个问题，相关的知识分散在多个文档/多个 chunk 里。单个 chunk 看都不完整，模型只能从每个 chunk 里摘一点，自己“拼凑”出一个答案——这一拼凑，就容易拼错。

解决思路也很直接：**让检索返回的内容更准、更全、更聚焦**。

### 项目检索引擎架构回顾

在项目文档 `RAG优化流程树与实现指南.md` 中，Ragent 的检索引擎设计了一套完整的多通道检索架构。这个架构的设计目标就是解决“召回不准、召回不全”的问题。

```
用户问题
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  检索引擎 (RetrievalEngine)                        │
│  ├─ 多通道并行检索 (MultiChannelRetrievalEngine)   │
│  │   ├─ IntentDirectedSearchChannel (意图定向)   │ → 精准定位知识库
│  │   ├─ KeywordSearchChannel (关键词)             │ → 补充精确匹配
│  │   └─ VectorGlobalSearchChannel (全局向量)      │ → 兜底高召回
│  └─ 后处理器链 (PostProcessor)                    │
│      ├─ DeduplicationPostProcessor (去重)         │ → 消除重复
│      └─ RerankPostProcessor (重排序)              │ → 提升相关性
└─────────────────────────────────────────────────────┘
    │
    ▼
返回 Top-K 相关知识
```

这个架构的核心理念是：**多通道互补 + 后处理精炼**。

### 多通道并行检索

单一检索方式的局限很明显：向量检索擅长语义但对精确关键词不敏感，关键词检索擅长精确匹配但无法理解语义。

多通道并行检索的思路是：**让多种检索方式同时工作，取长补短**。

> **这段伪代码做什么？**
>
> 这是多通道检索引擎的核心执行逻辑：
> 1. **并行执行**：多个检索通道同时启动，每个通道独立运行
> 2. **通道隔离**：单个通道失败不影响其他通道，通过 try-catch 隔离
> 3. **结果合并**：所有通道返回后，统一进入后处理阶段

```java
// 多通道并行检索 - 伪代码
public List<RetrievedChunk> multiChannelSearch(String query, 
                                                List<SubQuestionIntent> intents,
                                                int topK) {
    // 1. 构建检索上下文
    SearchContext context = buildContext(query, intents, topK);
    
    // 2. 获取所有启用的检索通道
    List<SearchChannel> channels = getEnabledChannels(context);
    
    // 3. 并行执行所有通道（CompletableFuture）
    List<SearchChannelResult> results = channels.parallelStream()
        .map(channel -> {
            try {
                return channel.search(context);  // 每个通道独立检索
            } catch (Exception e) {
                log.error("通道 {} 执行失败", channel.getName(), e);
                return emptyResult();  // 失败返回空结果，不影响其他通道
            }
        })
        .collect(Collectors.toList());
    
    // 4. 执行后处理器链：去重 → 重排序
    return postProcess(results, context);
}
```

### 意图定向检索：精准定位知识库

意图定向检索是 Ragent 检索引擎的核心创新。它的设计思想是：**先理解用户想问什么，再去对应的知识库精准查找**。

这比“在大海捞针”式的全局检索要精准得多。

> **这段伪代码做什么？**
>
> 这是意图定向检索的启用判断和执行逻辑：
> 1. **启用条件**：只有识别出 KB（知识库）类型的意图时才启用
> 2. **精准定位**：根据意图 ID 找到对应的知识库，只在该知识库中检索
> 3. **置信度传递**：检索结果的置信度 = 意图匹配的最高分数

```java
// 意图定向检索 - 伪代码
public boolean isEnabled(SearchContext context) {
    // 只有存在 KB 意图时才启用（排除系统类、MCP类意图）
    List<NodeScore> kbIntents = extractKbIntents(context);
    return CollUtil.isNotEmpty(kbIntents);
}

public SearchChannelResult search(SearchContext context) {
    // 1. 提取 KB 意图
    List<NodeScore> kbIntents = extractKbIntents(context);
    
    // 2. 根据每个意图精准检索对应知识库
    List<RetrievedChunk> allChunks = new ArrayList<>();
    for (NodeScore intent : kbIntents) {
        String collectionName = intent.getNode().getCollectionName();
        List<RetrievedChunk> chunks = milvus.search(
            collectionName, 
            context.getQuery(), 
            context.getTopK()
        );
        allChunks.addAll(chunks);
    }
    
    // 3. 置信度 = 意图匹配的最高分数
    double confidence = kbIntents.stream()
        .mapToDouble(NodeScore::getScore)
        .max()
        .orElse(0.0);
    
    return SearchChannelResult.builder()
        .channelType(INTENT_DIRECTED)
        .chunks(allChunks)
        .confidence(confidence)
        .build();
}
```

### 全局兜底检索：确保不漏检

意图定向检索很精准，但它的前提是“能识别出意图”。如果用户的问题比较模糊，或者意图识别置信度很低，意图检索可能失效。

这时候就需要全局向量检索来兜底——在所有知识库中做向量检索，确保不漏检。

> **这段伪代码做什么？**
>
> 这是全局兜底检索的触发逻辑：
> 1. **兜底条件**：没有识别出意图 OR 意图置信度 < 0.5
> 2. **全局检索**：在所有知识库中检索，召回率最高
> 3. **置信度中等**：因为是兜底方案，置信度设为 0.7

```java
// 全局兜底检索 - 伪代码
public boolean isEnabled(SearchContext context) {
    // 条件1：没有识别出任何意图
    List<NodeScore> allIntents = context.getAllIntents();
    if (CollUtil.isEmpty(allIntents)) {
        return true;  // 必须兜底
    }
    
    // 条件2：意图最高分 < 0.5（置信度不足）
    double maxScore = allIntents.stream()
        .mapToDouble(NodeScore::getScore)
        .max()
        .orElse(0.0);
    
    return maxScore < 0.5;  // 置信度不足，启用兜底
}

public SearchChannelResult search(SearchContext context) {
    // 获取所有 KB 类型的知识库
    List<String> allCollections = getAllKBCollections();
    
    // 并行在所有知识库中检索
    List<RetrievedChunk> allChunks = allCollections.parallelStream()
        .map(collection -> milvus.search(collection, context.getQuery(), topK))
        .flatMap(List::stream)
        .sorted(byScore())  // 按相似度排序
        .limit(context.getTopK())
        .collect(Collectors.toList());
    
    return SearchChannelResult.builder()
        .channelType(VECTOR_GLOBAL)
        .chunks(allChunks)
        .confidence(0.7)  // 兜底方案，置信度中等
        .build();
}
```

### 后处理器链：让结果更干净

多通道检索返回的结果可能有重复、可能排序不够精准。后处理器链的作用就是**对结果进行精炼**。

去重处理器确保同样的内容不会重复出现；重排序处理器（RerankPostProcessor）用更强的模型对候选结果重新打分，把最相关的排到前面。

> **这段伪代码做什么？**
>
> 这是后处理器链的执行逻辑：
> 1. **去重**：根据 chunk ID 或内容 hash 去重，保留分数最高的
> 2. **重排序**：调用 Rerank 模型对候选结果重新打分
> 3. **截断**：返回最终 Top-K

```java
// 后处理器链 - 伪代码
public List<RetrievedChunk> postProcess(List<SearchChannelResult> results,
                                         SearchContext context) {
    // 1. 合并所有通道的 chunks
    List<RetrievedChunk> merged = results.stream()
        .flatMap(r -> r.getChunks().stream())
        .collect(Collectors.toList());
    
    // 2. 去重：根据内容 hash 或 ID 去重，保留分数最高的
    Map<String, RetrievedChunk> deduped = new LinkedHashMap<>();
    for (RetrievedChunk chunk : merged) {
        String key = chunk.getId() != null ? chunk.getId() 
                     : String.valueOf(chunk.getContent().hashCode());
        if (!deduped.containsKey(key) 
            || chunk.getScore() > deduped.get(key).getScore()) {
            deduped.put(key, chunk);
        }
    }
    List<RetrievedChunk> dedupedChunks = new ArrayList<>(deduped.values());
    
    // 3. 重排序：调用 Rerank 模型
    if (dedupedChunks.size() > context.getTopK()) {
        return rerankService.rerank(context.getQuery(), 
                                    dedupedChunks, 
                                    context.getTopK());
    }
    
    return dedupedChunks;
}
```

### 检索优化对幻觉控制的价值

总结一下检索增强对幻觉控制的价值：

- **多通道互补**：向量检索 + 关键词检索 + 意图定向，覆盖更全面
- **意图定向**：精准定位知识库，减少无关内容的干扰
- **全局兜底**：确保不漏检，防止模型“被迫编造”
- **后处理精炼**：去重 + 重排序，让最终结果更聚焦

**核心观点**：好的检索让模型“有正确的料”，是幻觉控制的第一道防线。后续的 Prompt 约束、引用验证、事后校验，都是建立在这个基础之上的。

## 引用验证机制

Prompt 约束告诉模型“要用参考资料、要标引用”，但模型可能“敷衍了事”——引用了但内容不支持，或者没引用但其实用了。

引用验证要解决的就是这个问题：**检查模型输出的每个事实，是否真的能在参考资料中找到支持**。

### 什么是引用验证

引用验证的核心逻辑是：**对模型输出中的每个关键断言，检查它是否能对应到参考资料中的原文**。

具体做法是：

1. **解析输出**：从模型输出中提取所有带引用 `[n]` 的事实陈述
2. **匹配来源**：对每个引用，检查对应的参考资料是否真的支持这个陈述
3. **判定真伪**：如果参考资料不支持，标记为“可疑”，触发修正或降级

### 引用验证的实现思路

> **这段代码做什么？**
>
> 这是引用验证的核心实现逻辑：
> 1. **提取引用**：用正则从模型输出中提取 `[n]` 格式的引用编号
> 2. **匹配验证**：对每个引用，检查原文中是否包含该事实的关键信息
> 3. **结果判定**：如果验证失败（引用了但原文不支持），标记为不可信

```java
/**
 * 引用验证服务
 */
public class CitationVerificationService {

    /**
     * 验证模型输出中的引用是否有效
     *
     * @param modelOutput 模型生成的完整回答
     * @param chunks      参考资料列表
     * @return 验证结果
     */
    public VerificationResult verify(String modelOutput, List<RetrievedChunk> chunks) {
        // 1. 提取模型输出中的所有引用
        List<CitationWithStatement> citations = extractCitations(modelOutput);
        
        // 2. 逐个验证引用是否有效
        List<CitationCheckResult> results = new ArrayList<>();
        for (CitationWithStatement citation : citations) {
            CitationCheckResult checkResult = checkCitation(
                citation.getStatement(),    // 陈述内容
                citation.getCitationNum(),  // 引用编号
                chunks                      // 参考资料
            );
            results.add(checkResult);
        }
        
        // 3. 汇总结果
        return buildResult(results);
    }

    /**
     * 提取输出中的引用及其对应陈述
     * 例如： "...退货政策是 7 天[1] ..." -> 提取出 "退货政策是 7 天" 和引用编号 1
     */
    private List<CitationWithStatement> extractCitations(String output) {
        List<CitationWithStatement> citations = new ArrayList<>();
        
        // 正则匹配 [数字] 格式的引用
        Pattern pattern = Pattern.compile("([^\\[]+)\\[(\\d+)\\]");
        Matcher matcher = pattern.matcher(output);
        
        while (matcher.find()) {
            String statement = matcher.group(1).trim();
            int citationNum = Integer.parseInt(matcher.group(2));
            citations.add(new CitationWithStatement(statement, citationNum));
        }
        
        return citations;
    }

    /**
     * 验证单个引用是否有效
     * 检查陈述中的关键信息是否能在原文中找到
     */
    private CitationCheckResult checkCitation(String statement, 
                                              int citationNum,
                                              List<RetrievedChunk> chunks) {
        // 引用编号从 1 开始，数组从 0 开始
        int index = citationNum - 1;
        if (index < 0 || index >= chunks.size()) {
            return CitationCheckResult.invalid(
                statement, citationNum, 
                "引用编号超出范围"
            );
        }
        
        RetrievedChunk referencedChunk = chunks.get(index);
        String originalText = referencedChunk.getContent();
        
        // 检查陈述中的关键信息是否在原文中
        // 这里可以用关键词匹配，也可以用更复杂的语义匹配
        boolean isSupported = checkStatementSupported(statement, originalText);
        
        if (isSupported) {
            return CitationCheckResult.valid(statement, citationNum);
        } else {
            return CitationCheckResult.invalid(
                statement, citationNum,
                "原文不支持此陈述"
            );
        }
    }

    /**
     * 简单验证：检查陈述中的关键词是否在原文中
     * 实际项目中可以用更复杂的语义匹配
     */
    private boolean checkStatementSupported(String statement, String originalText) {
        // 提取陈述中的实词（简化处理：去掉停用词）
        List<String> keywords = extractKeywords(statement);
        
        // 检查关键词是否都在原文中
        for (String keyword : keywords) {
            if (!originalText.contains(keyword)) {
                return false;  // 有关键词不在原文中，可能有幻觉
            }
        }
        return true;
    }

    /**
     * 构建验证结果
     */
    private VerificationResult buildResult(List<CitationCheckResult> results) {
        long invalidCount = results.stream()
            .filter(r -> !r.isValid())
            .count();
        
        return VerificationResult.builder()
            .totalReferences(results.size())
            .validCount(results.size() - invalidCount)
            .invalidCount((int) invalidCount)
            .invalidCitations(results.stream()
                .filter(r -> !r.isValid())
                .collect(Collectors.toList()))
            .isPass(invalidCount == 0)
            .build();
    }
}
```

### 引用验证的效果

引用验证的价值不在于“找茬”，而在于：

- **发现幻觉**：快速定位可能的错误输出
- **触发修正**：验证失败时可以让模型重新回答，或降级为“我不确定”
- **评估质量**：统计引用覆盖率，作为系统质量的指标之一

但引用验证也有局限：它只能检查“有没有引用”以及“引用能不能对应上”，无法检查“事实本身对不对”。比如原文中写的是“7天无理由退货”，模型引用了 [1]，但把“7天”听成了“15天”——这种错误引用验证是看不出来的。

这就要靠后续的**多答案一致性检验**和**事后校验**来补充。

## 多答案一致性检验

### 自我一致性策略的原理

如果一个问题让模型回答三次，三次都给出同样的答案，那这个答案的可信度就高；如果三次答案各不相同，说明模型在“猜”。

这就是**自我一致性**（Self-Consistency）的核心思想：用采样或多次生成的方式，让模型产生多个答案，然后通过投票或其他方式选择最一致的作为最终答案。

### 实现思路

> **这段伪代码做什么？**
>
> 这是多答案一致性检验的实现逻辑：
> 1. **多次生成**：用相同的 Prompt 让模型生成多个答案（可以调高 temperature 增加多样性）
> 2. **答案提取**：从每个回答中提取核心结论
> 3. **投票选择**：选择出现次数最多的答案作为最终答案

```java
/**
 * 多答案一致性检验
 */
public class SelfConsistencyChecker {

    /**
     * 通过多次生成检验答案一致性
     *
     * @param question    用户问题
     * @param chunks      参考资料
     * @param iterations   生成次数（一般 3~5 次）
     * @param temperature  生成温度（可以设高一点增加多样性）
     * @return 一致性检验结果
     */
    public ConsistencyResult check(String question, 
                                    List<RetrievedChunk> chunks,
                                    int iterations,
                                    double temperature) {
        
        // 1. 多次生成答案
        List<String> answers = new ArrayList<>();
        for (int i = 0; i < iterations; i++) {
            String answer = llmService.chat(
                buildPrompt(question, chunks),
                temperature,  // 可以设高一点，让答案有多样性
                maxTokens
            );
            answers.add(extractCoreAnswer(answer));  // 提取核心结论
        }
        
        // 2. 统计答案出现次数
        Map<String, Long> frequency = answers.stream()
            .collect(Collectors.groupingBy(a -> a, Collectors.counting()));
        
        // 3. 找出最一致的答案
        String consensusAnswer = frequency.entrySet().stream()
            .max(Map.Entry.comparingByValue())
            .map(Map.Entry::getKey)
            .orElse(answers.get(0));
        
        // 4. 计算一致性得分
        double consistencyScore = (double) frequency.get(consensusAnswer) / iterations;
        
        return ConsensusResult.builder()
            .consensusAnswer(consensusAnswer)
            .consistencyScore(consistencyScore)
            .answerDistribution(frequency)
            .isConsistent(consistencyScore >= 0.6)  // 60% 以上一致视为可信
            .build();
    }

    /**
     * 从完整回答中提取核心结论（简化处理）
     */
    private String extractCoreAnswer(String fullAnswer) {
        // 简化：取回答的第一句话或第一段
        String[] sentences = fullAnswer.split("[。！？]");
        return sentences[0].trim();
    }
}
```

### 适用场景与局限性

多答案一致性检验适合的场景：

- **是非类问题**：答案比较确定的问题，如“7天无理由退货对不对”
- **需要高可信度**：对准确率要求高的场景，如医疗、法律

不适合的场景：

- **开放性问题**：答案本来就可能多种多样，如“如何评价这件事”
- **对延迟敏感**：多次生成会增加延迟

**局限**：自我一致性只能检测“答案是否一致”，不能检测“一致答案是否正确”。如果模型三次都答错了，检验也会认为“一致”。

## 事实性校验层：生成后的守护

即使前面的手段都用上了，幻觉还是可能发生。最后一道防线是**事后校验**：在模型生成答案之后，额外加一个检查环节。

### 事后校验的思路

事后校验的核心是：**让另一个模型（或同一个模型）来检查生成的答案是否可信**。

具体做法：

1. **提取事实**：从生成的答案中提取关键断言
2. **逐条校验**：每个断言都回去查原始资料，看是否支持
3. **判定结果**：如果有不支持的，标记为“可疑”，触发处理

### 校验模型的使用方式

校验可以用两种方式：

**方式一：同一个模型，先答后检**。让模型先给出答案，然后让模型自己检查自己的答案。

```
用户问题：XXX

请你回答上述问题。

---

现在请检查你刚才的回答是否正确：
1. 回答中的每个事实是否都能在参考资料中找到？
2. 回答中有没有参考资料不支持的信息？
3. 如果有问题，请指出并给出修正后的答案。
```

**方式二：独立的校验模型**。用一个更小、更专注的模型来做校验，专门检查事实性。

校验模型的好处是：可以更聚焦、更可控，不容易被模型“带偏”。

### 校验结果的处理

校验失败后，系统可以选择：

- **重新生成**：让模型基于校验反馈重新回答
- **降级回复**：如果校验失败次数过多，降级为“我不确定”
- **标记输出**：不拦截，但给用户一个可信度提示

## 总结与实践建议

### 核心要点回顾

这篇文档按照“源头到末端”的顺序，讲解了 RAG 系统中的大模型生成策略和幻觉控制手段：

1. **生成参数**：temperature = 0~0.3，追求准确性而非创意
2. **Prompt 约束**：明确告诉模型“只能用参考资料”“不要编造”
3. **检索增强**：好的检索是幻觉控制的第一道防线
4. **引用验证**：检查模型输出是否真的“有据可依”
5. **多答案一致性**：用冗余检测来排除错误答案
6. **事后校验**：生成后的兜底防线

### 技术选型建议

这些手段不用全上，根据场景选择：

| 场景 | 推荐策略 |
|------|----------|
| 对准确率要求一般 | Prompt 约束 + 低 temperature |
| 对准确率要求较高 | + 检索增强 + 引用验证 |
| 对准确率要求极高 | + 多答案一致性 + 事后校验 |
| 对延迟敏感 | 优先 Prompt 约束，谨慎加其他层 |
| 对成本敏感 | 检索增强（收益比最高） |

### 演进路径

建议按这个顺序逐步升级：

1. **第一步**：先调好 Prompt + temperature，这是零成本立刻见效的
2. **第二步**：优化检索质量，确保检索回来的知识又准又全
3. **第三步**：加引用验证，发现问题及时拦截
4. **第四步**：对关键问题加一致性检验和事后校验

---

幻觉控制是一个系统工程，没有银弹。每加一层手段，都能拦截一部分幻觉，但也会增加延迟和成本。找到适合自己场景的平衡点，比盲目堆砌手段更重要。
