# RAG 系列文档写作方法论

本文档总结编写 RAG 系列文档（如《大模型生成策略与幻觉控制》）时的方法论和规范，供后续文档编写时参照使用。

---

## 一、文档设计方法论

### 1.1 核心设计原则

**平衡型深度**：主要技术点讲透，次要技术点简要提及，适合系统学习。

**开篇引入**：从一个具体场景或痛点切入，让读者产生共鸣。

**逻辑递进**：按照“背景→原理→实践→代码→总结”的顺序渐进展开。

**承前启后**：适当引用前文内容，形成知识体系。

### 1.2 文档结构模板

一篇完整的文档通常包含以下结构：

```
1. 开篇引入（500-800字）
   - 场景痛点
   - 核心概念
   - 本篇内容概览

2. 核心概念与原理（1500-2500字）
   - 是什么、为什么
   - 核心要素解析
   - 对比/对比表

3. 技术实现（1500-2500字）【重点章节，有代码/伪代码】
   - 实现思路
   - 代码前讲解（"这段代码做什么"）
   - 代码/伪代码示例
   - 代码后分析

4. 实践与应用（800-1500字）
   - 场景选择
   - 效果对比
   - 选型建议

5. 总结与展望（300-500字）
   - 核心要点回顾
   - 技术选型建议
   - 未来趋势（可选）
```

### 1.3 章节规划方法

**确定目标读者**：明确文档服务谁（Java开发者？架构师？）

**确定文档定位**：
- overview 文档（3000-5000字）：快速扫盲
- 系统学习文档（8000-12000字）：全面覆盖
- 深度专题文档（15000字+）：事无巨细

**规划章节权重**：
- 重点章节（2-3个）：详细讲解，配完整代码
- 次要章节（1-2个）：概念讲解，配代码片段
- 过渡章节（1个）：承上启下，概念为主

## 二、写作风格规范

### 2.1 语言风格

**简洁直接**：能用一句话说清楚的不用两句，避免废话和车轱辘话。

**口语化但不随意**：像在和读者对话，但保持专业性。

**少用术语解释**：术语首次出现时自然带出定义，后续不再重复解释。

**示例对比**：用对比表格、对比代码来展示差异，而非用描述性文字。

### 2.2 结构化表达

**层次分明**：用多级标题组织内容层级。

**关键信息加粗**：突出核心概念、参数名、重要结论。

**代码块清晰**：代码有标题、有注释、有运行效果说明。

**表格辅助**：对比类内容用表格，一目了然。

### 2.3 专业性体现

**引用来源**：技术细节标出来源（如"根据 Milvus 官方文档"）。

**数据支撑**：关键结论有数据或案例支撑。

**工程视角**：不仅讲原理，还讲实际开发中的坑和经验。

## 三、代码示例处理规范

### 3.1 代码分级

| 级别 | 说明 | 格式 |
|------|------|------|
| L1 | 完整可运行的代码，零门槛复制使用 | 完整 Java 类/方法 |
| L2 | 核心逻辑代码，配合讲解使用 | 代码片段 + 上下文说明 |
| L3 | 伪代码，展示实现思路 | 伪代码 + 详细讲解 |

**选择原则**：
- 核心章节（2-3个）：用 L1 级别代码
- 次要章节：用 L2/L3 级别代码
- 概念章节：纯文字，无需代码

### 3.2 代码前讲解规范

每个代码块（无论级别）前必须有"这段代码做什么"的讲解：

> **这段代码做什么？**
>
> 这是 XXX 的核心实现逻辑：
> 1. **步骤一**：做了一件什么事
> 2. **步骤二**：做了什么处理
> 3. **步骤三**：返回什么结果

讲解要包含：
- 代码的作用（what）
- 代码的执行流程（how）
- 关键参数/变量的含义

### 3.3 代码格式规范

```java
/**
 * 类/方法的功能说明
 */
public class ExampleClass {

    // 常量/配置说明
    private static final int DEFAULT_TOP_K = 10;

    /**
     * 方法的功能说明
     *
     * @param param1 参数1的含义
     * @param param2 参数2的含义
     * @return 返回值的含义
     */
    public ReturnType methodName(Type1 param1, Type2 param2) {
        // 关键逻辑有中文注释
        // 核心步骤说明
        return result;
    }
}
```

### 3.4 伪代码规范

当不需要完整代码但需要展示逻辑时，使用伪代码：

> **这段伪代码做什么？**
>
> 这是 XXX 的执行流程：
> 1. 第一步做什么
> 2. 第二步做什么
> 3. 返回什么

```java
// 伪代码格式示例
public Result process(Input input) {
    // 1. 参数校验
    if (input == null) {
        return emptyResult();
    }
    
    // 2. 核心处理
    List<Item> items = input.getItems();
    for (Item item : items) {
        // 处理逻辑
    }
    
    // 3. 返回结果
    return Result.builder()
        .items(processed)
        .count(processed.size())
        .build();
}
```

## 四、与项目文档结合规范

### 4.1 何时引用项目文档

**需要引用的情况**：
- 讲解某个技术的实际实现时
- 需要展示项目中的代码示例时
- 需要引用项目中的架构设计时

**引用格式**：
```
> 本节参考 `知识文档/RAG优化流程树与实现指南.md` 第4节

// 引用代码
// MultiChannelRetrievalEngine.java:66-90
```

### 4.2 项目文档引用规范

**引用位置**：
- 章节开头：说明本节内容与项目文档的关系
- 代码前：说明代码来自项目的哪个文件
- 架构图旁：说明架构设计来自项目文档

**引用深度**：
- 简要引用：只说明"详见项目文档 XXX"
- 详细引用：重复核心代码并加讲解

### 4.3 伪代码的使用场景

使用伪代码代替完整代码的场景：
- 项目代码过长，完整展示会冲淡重点
- 项目代码涉及多个模块，不便拆开讲解
- 需要展示跨模块的协作逻辑
- 纯概念性内容，不需要实际运行代码

## 五、内容规划检查清单

写文档前，先完成以下规划：

### 5.1 目标确认
- [ ] 明确文档目标：让读者获得什么
- [ ] 明确读者定位：谁会看这个文档
- [ ] 明确文档定位：overview/系统学习/深度专题

### 5.2 结构规划
- [ ] 规划章节数量（通常 6-8 章）
- [ ] 划分章节权重（重点/次要/过渡）
- [ ] 确认哪些章节需要代码、什么级别的代码

### 5.3 内容准备
- [ ] 收集必要的背景知识
- [ ] 确认需要引用的前文/项目文档
- [ ] 准备代码示例（自己的示例或引用项目）

---

## 六、文档质量自检清单

写完后，检查以下方面：

### 6.1 结构检查
- [ ] 开篇是否有场景/痛点引入
- [ ] 各章节是否有清晰的层次
- [ ] 结尾是否有总结

### 6.2 内容检查
- [ ] 专业术语是否一致
- [ ] 概念解释是否准确
- [ ] 是否与前文形成呼应

### 6.3 代码检查
- [ ] 重点章节是否有代码
- [ ] 代码前是否有"这段代码做什么"讲解
- [ ] 代码后是否有效果/分析说明

### 6.4 格式检查
- [ ] 标题层级是否清晰
- [ ] 关键信息是否加粗
- [ ] 表格/图片是否规范

1. 参照本文档的设计方法论
2. 参考已完成的文档风格
3. 结合项目文档中的实际实现
4. 保持系列文档的一致性
