# 模型检索生成评估与优化

> 在 RAG 系统中，检索和生成的质量直接决定了最终的回答效果。但如何衡量这个“效果”？如何发现系统中的薄弱环节并进行针对性优化？这是本章要解决的问题。本文档将系统梳理 RAG 评估体系，并结合 Ragent 项目中的实际实现，讲解评估指标、评测方法和优化策略。

---

## 一、开篇引入：为什么 RAG 系统需要评估

### 1.1 一个真实的痛点

假设你上线了一个企业客服 RAG 系统，最初运行时效果还不错。但运行一段时间后，你发现：

- 用户开始抱怨“答非所问”
- 运营团队反馈某些知识点的召回率明显下降
- 开发者不知道是检索出了问题还是生成出了问题

这时你会发现：**你甚至无法准确定位问题出在哪里**。因为 RAG 是一个多环节系统，任何一个环节出问题都会影响最终效果，但没有评估体系，你只能靠主观感受来判断系统好坏。

这就是缺乏评估体系的代价——系统运行在“隐性盲区”中，问题积累到爆发时已经难以挽回。

### 1.2 评估的核心价值

RAG 评估体系的核心价值体现在三个层面：

**定位问题**：通过分解评估指标，精准定位是检索环节还是生成环节出了问题。

**指导优化**：通过持续监控评估指标，量化优化措施的效果，判断优化方向是否正确。

**保障质量**：通过建立评估基线，防止系统效果随着迭代而退化。

没有评估的 RAG 系统就像没有仪表盘的汽车——能开，但不知道油量、速度、发动机状态，随时可能抛锚。

### 1.3 本篇内容概览

本篇将从以下方面展开：

- **核心评估指标**：检索指标（Recall、Precision、MRR、nDCG）、生成指标（引用准确率、幻觉率）
- **评测方法**：离线评测、在线评测、A/B 测试
- **Ragent 评估体系实现**：项目中的评估模块设计
- **优化策略**：基于评估结果的针对性优化方法

## 二、核心评估指标体系

### 2.1 检索阶段评估指标

检索阶段的目标是**找到所有相关的内容**，评估指标围绕“找得全不全、找得准不准”来设计。

#### 2.1.1 Recall@K（召回率）

**定义**：在检索返回的 Top-K 结果中，包含的相关 Chunk 数量占全部相关 Chunk 数量的比例。

**计算公式**：
```
Recall@K = |{相关Chunk} ∩ {Top-K返回结果}| / |{全部相关Chunk}|
```

**业务含义**：Recall@K 衡量的是“找得全不全”。如果你的知识库中有 10 条和“退货政策”相关的文档，检索系统返回了 Top-20，其中包含了 8 条相关文档，那么 Recall@20 = 8/10 = 80%。

**Ragent 中的应用**：`MultiChannelRetrievalEngine` 返回的 `SearchChannelResult` 中记录了每个通道的召回情况，通过分析不同通道的 Recall@K 可以判断是否存在漏召回。

#### 2.1.2 Precision@K（精确率）

**定义**：在检索返回的 Top-K 结果中，相关 Chunk 所占的比例。

**计算公式**：
```
Precision@K = |{相关Chunk} ∩ {Top-K返回结果}| / K
```

**业务含义**：Precision@K 衡量的是“找得准不准”。如果检索返回了 Top-10，其中 7 条是相关的，那么 Precision@10 = 7/10 = 70%。

**注意**：Recall 和 Precision 通常是互斥的——追求高召回可能会拉低精确率，反之亦然。在 RAG 场景中，通常优先保证 Recall（宁可多召回也不能漏掉），然后在重排序阶段提升 Precision。

#### 2.1.3 MRR（平均倒数排名）

**定义**：第一个相关结果排名的倒数求平均值。

**计算公式**：
```
MRR = (1/N) × Σ(1/rank_i)
```
其中 rank_i 是第 i 个查询的第一个相关结果的排名。

**业务含义**：MRR 衡量的是“最重要的相关结果排得够不够靠前”。如果第一个相关结果排在第 5 位，那么这一项的得分是 1/5 = 0.2。

**Ragent 中的应用**：`IntentDirectedSearchChannel` 返回的结果按置信度排序，MRR 可以评估意图定向检索是否把最相关的知识库放在了前面。

#### 2.1.4 nDCG@K（归一化折损累积增益）

**定义**：对检索结果按相关性分档，计算 DCG 并归一化。

**计算公式**：
```
DCG@K = Σ(rel_i / log2(i+1))  // i 从 1 开始
nDCG@K = DCG@K / IDCG@K
```
其中 rel_i 是第 i 个结果的相关性得分（通常按 0、1、2、3 分档），IDCG 是理想情况下的 DCG。

**业务含义**：nDCG 考虑了不同结果的相关性差异——排在前面的相关结果比排在后面的更有价值。如果最相关的结果排在了第 10 位，即使召回率很高，nDCG 也会很低。

**Ragent 中的应用**：在 `RerankPostProcessor` 之后，nDCG@K 可以评估重排序的效果是否达到了预期。

#### 2.1.5 检索指标对比表

| 指标 | 衡量维度 | 适用场景 | RAG 优化方向 |
|------|----------|----------|---------------|
| Recall@K | 找得全不全 | 知识库内容多且不允许漏检的场景 | 增加检索通道、优化 Top-K |
| Precision@K | 找得准不准 | 对噪声敏感、上下文窗口有限的场景 | 优化重排序、提升去重效果 |
| MRR | 首个结果质量 | 特别关注 Top-1 质量的场景 | 优化意图识别、优化排序 |
| nDCG@K | 整体排序质量 | 综合评估排序效果的场景 | 优化 Reranker、优化融合策略 |

### 2.2 生成阶段评估指标

生成阶段的目标是**基于检索结果生成准确、可靠的回答**，评估指标围绕“答案对不对、答案好不好”来设计。

#### 2.2.1 引用准确率（Citation Accuracy）

**定义**：模型生成的答案中，正确引用检索结果的比例。

**计算公式**：
```
引用准确率 = 正确引用的 Chunk 数 / 总共引用的 Chunk 数
```

**业务含义**：如果模型回答中引用了 5 个 Chunk，其中有 4 个 Chunk 的内容确实能支撑回答中的对应观点，那么引用准确率 = 4/5 = 80%。

**Ragent 中的应用**：在 `RAGPromptService` 组装 Prompt 时，Chunk 被编号注入上下文，引用准确率衡量的是“模型有没有张冠李戴”。

#### 2.2.2 幻觉率（Hallucination Rate）

**定义**：模型生成的答案中，包含但无法从检索结果中验证的信息占比。

**计算公式**：
```
幻觉率 = 无法验证的陈述数 / 总陈述数
```

**业务含义**：如果模型生成了 10 句陈述，其中有 3 句无法从检索结果中找到依据，那么幻觉率 = 3/10 = 30%。

**为什么重要**：RAG 本意是用检索结果“约束”生成，但幻觉率衡量的是这个约束是否生效。如果幻觉率很高，说明即使检索到了相关内容，模型仍然在“编造”答案。

#### 2.2.3 答案相关性（Answer Relevance）

**定义**：生成的答案与用户问题的相关程度。

**评估方式**：通常需要人工标注或使用 LLM 作为评判模型（Judge Model）。

**Ragent 中的应用**：可以通过设计 Prompt 让 LLM 评判“回答是否切题、是否完整、是否有价值”。

#### 2.2.4 生成指标对比表

| 指标 | 衡量维度 | 评估方式 | RAG 优化方向 |
|------|----------|----------|---------------|
| 引用准确率 | 引用是否正确 | 自动化比对 Chunk 内容 | 优化 Prompt 模板、约束引用格式 |
| 幻觉率 | 是否编造内容 | 自动化比对 + 人工抽检 | 增强检索结果质量、调整生成温度 |
| 答案相关性 | 回答是否切题 | 人工 / LLM 评判 | 优化 Prompt、意图识别 |

### 2.3 端到端评估指标

除了分解的检索指标和生成指标，还需要从用户体验角度评估端到端效果。

#### 2.3.1 人工标注准确率

**定义**：通过人工抽样评估，统计模型正确回答问题的比例。

**评估标准**：
- 完全正确：答案准确引用了相关 Chunk，逻辑完整
- 部分正确：答案方向对但有细节错误
- 答非所问：答案与问题无关
- 无法回答：检索结果为空或不足以回答问题

**Ragent 中的应用**：定期抽样评估是金标准，可以校验自动化指标的有效性。

#### 2.3.2 用户追问率

**定义**：用户收到回答后继续提问的比例。

**业务含义**：追问率高通常说明首次回答没有解决用户问题，可能存在检索遗漏或答案不准确的问题。

#### 2.3.3 答非所问率

**定义**：用户标记回答与问题无关的比例。

**业务含义**：这是最直接的用户体验指标，反映系统是否正确理解了用户意图。

## 三、评测方法与实践

### 3.1 离线评测

**核心思想**：基于预先标注好的测试集，自动化评估系统效果。

**测试集构建**：
1. **准备 Query 集合**：覆盖不同类型、不同难度的问题
2. **准备 Ground Truth**：人工标注每个 Query 对应的相关 Chunk
3. **准备参考问答**：人工标注每个 Query 的理想答案

**评测流程**：
```
Query → RAG系统 → 检索结果 + 生成答案
         ↓
    与 Ground Truth 比对 → 计算指标
```

**Ragent 中的实践**：`MultiChannelRetrievalEngine` 可以配置输出详细的检索日志，便于离线分析 Recall@K 和 Precision@K。

> 本节参考 `知识文档/RAG优化流程树与实现指南.md` 第 2 节检索引擎相关代码

### 3.2 在线评测

**核心思想**：在生产环境中收集真实用户反馈，持续监控系统效果。

**数据采集**：
- 用户对回答的评分（点赞/点踩）
- 用户追问行为
- 用户是否标记了“答非所问”

**Ragent 中的实践**：通过 `@RagTraceNode` 注解记录每一次请求的完整链路，便于在线分析各环节的耗时和效果。

> 本节参考 `知识文档/RAG优化流程树与实现指南.md` 第 3 节工程级优化汇总

### 3.3 A/B 测试

**核心思想**：对比两个版本的系统效果，判断优化措施是否有效。

**适用场景**：
- 上线新的检索策略
- 更换 Reranker 模型
- 调整 Prompt 模板

**Ragent 中的实践**：`ModelRoutingExecutor` 支持配置多个模型候选，可以轻松实现模型级别的 A/B 测试。

### 3.4 评测方法对比

| 方法 | 优点 | 缺点 | 适用阶段 |
|------|------|------|----------|
| 离线评测 | 可重复、成本低、可自动化 | 需要人工标注测试集 | 迭代开发期 |
| 在线评测 | 反映真实场景 | 需要一定流量积累 | 上线后持续监控 |
| A/B 测试 | 量化优化效果 | 需要流量分配能力 | 优化措施验证 |

## 四、Ragent 评估体系实现

### 4.1 评估模块架构

Ragent 项目中的评估体系主要包含以下核心组件：

| 组件 | 职责 | 核心类 |
|------|------|--------|
| `RagTraceService` | 记录全链路调用数据 | `RagTraceServiceImpl` |
| `RagTraceQueryService` | 查询历史调用记录 | `RagTraceQueryService` |
| 评估指标计算 | 基于调用数据计算指标 | 各环节的日志分析 |

### 4.2 全链路追踪设计

Ragent 通过 `@RagTraceNode` 注解实现全链路追踪，每个 RAG 环节的执行都被记录：

```java
// IntentResolver.java:52-144 - 意图识别核心逻辑
@RagTraceNode(name = "intent-resolve", type = "INTENT")
public List<SubQuestionIntent> resolve(RewriteResult rewriteResult) {
    // 提取子问题列表
    List<String> subQuestions = CollUtil.isNotEmpty(rewriteResult.subQuestions())
            ? rewriteResult.subQuestions()
            : List.of(rewriteResult.rewrittenQuestion());
    
    // 并行对每个子问题进行意图分类 - 提升响应速度
    List<CompletableFuture<SubQuestionIntent>> tasks = subQuestions.stream()
            .map(q -> CompletableFuture.supplyAsync(
                    () -> new SubQuestionIntent(q, classifyIntents(q)),
                    intentClassifyExecutor
            ))
            .toList();
    
    List<SubQuestionIntent> subIntents = tasks.stream()
            .map(CompletableFuture::join)
            .toList();
    
    // 限制总意图数量（保底 + 配额策略）
    return capTotalIntents(subIntents);
}
```

> **这段代码做什么？**
>
> 这是意图识别环节的全链路追踪实现：
> 1. **链路标记**：`@RagTraceNode` 注解标记这是“intent-resolve”环节
> 2. **输入记录**：rewriteResult 作为输入被记录
> 3. **并行处理**：每个子问题并行分类，提升响应速度
> 4. **输出记录**：subIntents 作为输出被记录，后续可用于评估意图识别的准确率

**追踪的信息包括**：
- 每个环节的开始/结束时间
- 输入参数（问题改写结果、意图识别结果、检索结果等）
- 输出结果（生成的答案、引用等）

### 4.3 检索质量评估实现

基于追踪数据，可以分析检索环节的质量：

```java
// MultiChannelRetrievalEngine.java:66-162 - 多通道检索引擎核心实现
public List<RetrievedChunk> retrieveKnowledgeChannels(List<SubQuestionIntent> subIntents, int topK) {
    // 1. 构建检索上下文
    SearchContext context = buildSearchContext(subIntents, topK);
    
    // 【阶段1：多通道并行检索】
    List<SearchChannelResult> channelResults = executeSearchChannels(context);
    if (CollUtil.isEmpty(channelResults)) {
        return List.of();
    }

    // 【阶段2：后置处理器链处理】
    return executePostProcessors(channelResults, context);
}

// 并行执行所有检索通道
private List<SearchChannelResult> executeSearchChannels(SearchContext context) {
    // 过滤启用的通道并按优先级排序
    List<SearchChannel> enabledChannels = searchChannels.stream()
            .filter(channel -> channel.isEnabled(context))
            .sorted(Comparator.comparingInt(SearchChannel::getPriority))
            .toList();

    log.info("启用的检索通道：{}", enabledChannels.stream().map(SearchChannel::getName).toList());

    // 并行执行 - 每个通道独立线程池
    List<CompletableFuture<SearchChannelResult>> futures = enabledChannels.stream()
            .map(channel -> CompletableFuture.supplyAsync(() -> {
                try {
                    return channel.search(context);
                } catch (Exception e) {
                    log.error("检索通道 {} 执行失败", channel.getName(), e);
                    return SearchChannelResult.builder()
                            .channelType(channel.getType())
                            .channelName(channel.getName())
                            .chunks(List.of())
                            .confidence(0.0)
                            .build();
                }
            }, ragRetrievalExecutor))
            .toList();

    // 等待所有结果并统计
    return futures.stream()
            .map(CompletableFuture::join)
            .filter(Objects::nonNull)
            .toList();
}
```

> **这段代码做什么？**
>
> 这是多通道检索引擎的实现，用于支持检索质量评估：
> 1. **通道隔离**：每个检索通道独立执行，单个通道失败不影响其他通道
> 2. **结果记录**：每个通道返回 `SearchChannelResult`，包含通道类型、名称、Chunk 列表、置信度、耗时
> 3. **后处理链**：检索结果经过去重、重排序等后处理，最终返回 Top-K
> 4. **评估支撑**：通过分析各通道的召回情况，可以评估不同检索策略的效果差异

### 4.4 评估指标计算示例

基于追踪数据计算检索指标：

```java
// 评估指标计算伪代码
public class RetrievalMetricsCalculator {
    
    // 计算 Recall@K
    public double calculateRecallAtK(List<RetrievedChunk> retrieved, 
                                     Set<String> relevantChunkIds, 
                                     int k) {
        List<RetrievedChunk> topK = retrieved.stream().limit(k).toList();
        long hitCount = topK.stream()
                .filter(chunk -> relevantChunkIds.contains(chunk.getId()))
                .count();
        return (double) hitCount / relevantChunkIds.size();
    }
    
    // 计算 Precision@K
    public double calculatePrecisionAtK(List<RetrievedChunk> retrieved, 
                                        Set<String> relevantChunkIds, 
                                        int k) {
        List<RetrievedChunk> topK = retrieved.stream().limit(k).toList();
        long hitCount = topK.stream()
                .filter(chunk -> relevantChunkIds.contains(chunk.getId()))
                .count();
        return (double) hitCount / k;
    }
    
    // 计算 MRR
    public double calculateMRR(List<RetrievedChunk> retrieved, 
                                Set<String> relevantChunkIds) {
        for (int i = 0; i < retrieved.size(); i++) {
            if (relevantChunkIds.contains(retrieved.get(i).getId())) {
                return 1.0 / (i + 1);
            }
        }
        return 0.0;
    }
}
```

## 五、基于评估的优化策略

### 5.1 检索优化

#### 5.1.1 召回率低时的优化

**问题表现**：Recall@K 低于预期，相关 Chunk 没有被召回。

**排查思路**：
1. 检查知识库是否包含相关内容
2. 检查 Embedding 模型是否适用于当前领域
3. 检查查询改写是否丢失了关键信息
4. 检查是否存在检索通道漏检

**优化措施**：
- **补充知识库**：如果是知识库缺失，补充相关文档
- **优化查询改写**：`QueryRewriteService` 可以增强术语归一化能力
- **增加检索通道**：`MultiChannelRetrievalEngine` 可以增加关键词检索通道

> 本节参考 `知识文档/应用篇/向量检索及召回优化.md` 第 5 节混合检索相关代码

#### 5.1.2 排序不理想时的优化

**问题表现**：nDCG@K 低，最相关的结果没有排在前面。

**优化措施**：
- **启用重排序**：`RerankPostProcessor` 可以显著提升排序质量
- **优化 RRF 融合参数**：调整 `rrfK` 参数
- **增加 Reranker 候选数**：从 20 提升到 30 或 50

### 5.2 生成优化

#### 5.2.1 幻觉率高时的优化

**问题表现**：模型生成的内容无法从检索结果中验证。

**优化措施**：
- **约束 Prompt**：在 Prompt 中明确要求“只基于提供的上下文回答”
- **降低生成温度**：将 temperature 调低（从 0.3 降到 0.1）
- **增强引用约束**：要求模型在回答中标注引用来源

> 本节参考 `知识文档/应用篇/大模型生成策略及幻觉控制.md`

#### 5.2.2 引用不准确时的优化

**问题表现**：模型引用了错误的 Chunk 编号。

**优化措施**：
- **固定引用格式**：使用 `[引用序号]` 格式，并在 Prompt 中强调
- **减少候选数量**：降低 Top-K，减少模型混淆的可能
- **增强上下文清晰度**：在上下文格式化时增加 Chunk 序号标注

### 5.3 端到端优化

#### 5.3.1 人工标注准确率低的优化

**优化流程**：
1. 拆解问题：分析是检索问题还是生成问题
2. 分环节优化：检索问题按 5.1 优化，生成问题按 5.2 优化
3. 验证效果：通过离线评测确认优化有效后再上线

#### 5.3.2 用户追问率高的优化

**优化思路**：
1. 分析追问的典型模式
2. 检查是否存在意图识别错误
3. 检查是否存在检索遗漏
4. 优化 Prompt 让答案更完整

## 六、总结与展望

### 6.1 核心要点回顾

1. **评估是 RAG 系统的基础设施**：没有评估就无法定位问题、优化效果、保障质量
2. **检索指标关注“找得全不全、找得准不准”**：Recall@K、Precision@K、MRR、nDCG@K 是核心指标
3. **生成指标关注“答案对不对、答案好不好”**：引用准确率、幻觉率、答案相关性是关键
4. **Ragent 提供了全链路追踪能力**：通过 `@RagTraceNode` 注解记录每个环节的输入输出
5. **优化需要基于评估数据**：先定位问题，再针对性优化，最后验证效果

### 6.2 技术选型建议

| 场景 | 推荐评估方案 |
|------|-------------|
| 迭代开发期 | 离线评测 + 测试集循环评估 |
| 上线初期 | 离线评测 + 在线小流量监控 |
| 稳定运营 | 在线评测 + 定期人工抽检 + A/B 测试 |

### 6.3 未来趋势

1. **自动化评估**：使用 LLM 作为 Judge Model，自动评估生成质量
2. **实时指标看板**：在线实时展示关键指标，监控系统健康度
3. **智能优化建议**：基于评估数据自动给出优化建议
