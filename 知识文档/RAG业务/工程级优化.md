# Ragent 工程级优化详解

> 本文档作为 RAG 优化流程树与实现指南的工程篇，覆盖限流与并发、全链路可观测性、MCP 工具生态三大企业级工程优化模块，配以代码示例和核心类说明。

## 一、限流与并发

### 整体流程：

```
用户 SSE 请求进入
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ @ChatRateLimit 注解切面 (ChatRateLimitAspect)                               │
│    ├── 提取参数：question, conversationId, SseEmitter                        │
│    └── 调用 ChatQueueLimiter.enqueue()                                       │
└─────────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ 分布式信号量排队 (ChatQueueLimiter)                                          │
│    ├── 请求入队（Redis SortedSet 按全局序号排序）                            │
│    ├── 尝试获取信号量（tryAcquirePermit）                                    │
│    │   ├── 获取成功 → 立即提交业务逻辑到线程池                               │
│    │   └── 获取失败 → 注册定时轮询 + 订阅发布通知                            │
│    └── SSE 连接关闭时自动释放信号量 + 发布通知唤醒等待者                     │
└─────────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ 业务逻辑执行 (RAGChatServiceImpl.streamChat)                                 │
│    └── 完成后 emitter.onCompletion 触发信号量释放                            │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 核心类：

| 类名 | 职责 |
|------|------|
| `@ChatRateLimit` | 标注需要限流的 SSE 入口方法 |
| `ChatRateLimitAspect` | AOP 切面，拦截注解方法，提取参数，调用限流器 |
| `ChatQueueLimiter` | 核心限流器：Redis 信号量 + 全局 FIFO 队列 + 发布-订阅通知 |
| `RAGRateLimitProperties` | 限流配置（最大并发数、队列超时、租约时长） |

### 优化点：

| 优化项 | 说明 | 核心代码位置 |
|--------|------|--------------|
| 全局信号量 | Redis PermitExpirableSemaphore，分布式部署下全局限流 | `SEMAPHORE_NAME = "rag:global:chat"` |
| FIFO 排队 | Redis SortedSet 按全局递增序号排序，先到先得 | `QUEUE_KEY = "rag:global:chat:queue"` |
| Lua 原子抢占 | Lua 脚本原子判断队列位置 + 移除操作，防止并发超发 | `lua/queue_claim_atomic.lua` |
| 发布-订阅唤醒 | 信号量释放后发布通知，等待者无需空轮询，响应更快 | `NOTIFY_TOPIC = "rag:global:chat:queue:notify"` |
| 自动 Lease 过期 | 信号量持有有 TTL，连接异常时自动释放，防死锁 | `globalLeaseSeconds` 配置项 |
| 超时优雅拒绝 | 队列等待超时后发送 SSE 拒绝消息并记录对话历史 | `recordRejectedConversation()` |
| TTL 跨线程透传 | `TtlExecutors` 包装线程池，用户 ID 等上下文自动传递 | `chatEntryExecutor` 配置 |

**代码示例：**

> **业务例子（伪代码）**
>
> ```
> 配置：globalMaxConcurrent=3，globalMaxWaitSeconds=30
>
> 时间轴（并发10个用户同时发问）：
>
>   T=0s：用户A/B/C 同时到达
>   - 用户A：入队 seq=1 → tryAcquirePermit() → 获得 permit-1 → 立即执行
>   - 用户B：入队 seq=2 → tryAcquirePermit() → 获得 permit-2 → 立即执行
>   - 用户C：入队 seq=3 → tryAcquirePermit() → 获得 permit-3 → 立即执行
>   （信号量已全部占满，availablePermits=0）
>
>   T=0s：用户D/E/F 相继到达（信号量已满）
>   - 用户D：入队 seq=4 → tryAcquirePermit() → 失败 → 注册轮询 + 订阅通知，等待
>   - 用户E：入队 seq=5 → 等待
>   - 用户F：入队 seq=6 → 等待
>
>   T=5s：用户A 回答完成，SSE 关闭
>   - emitter.onCompletion → releaseOnce()
>   - 释放 permit-1，publishQueueNotify()
>
>   T=5s：PollNotifier 收到 "permit_released" 通知
>   - 调用 Lua 脚本：claimIfReady(requestId=D的ID, availablePermits=1)
>   - Lua：D 是 SortedSet 中 score 最小（最早入队）的请求 → 移除并返回 claimed=1
>   - 用户D：tryAcquirePermit() → 获得信号量 → 立即提交业务逻辑
>
>   T=31s：用户F（等待超过30秒）
>   - deadline = 0s + 30s = 30s，现在 31s > 30s
>   - 从队列移除，publishQueueNotify()（让后面的人可能有机会）
>   - sendRejectEvents(emitter)：发送 SSE 事件
>     → event: reject，data: "系统繁忙，请稍后再试"
>     → event: done
>   - recordRejectedConversation()：写入用户消息 + 拒绝回复到对话历史
> ```
>
> **这段代码做什么？**
>
> 这是"全局 SSE 并发限流与排队"的核心实现，保证服务在高并发下不过载：
> 1. **全局信号量**：无论多少实例部署，Redis 信号量保证全局最多 N 个并发（如3个），不是每个实例各3个
> 2. **FIFO 排队**：超出并发上限的请求按先后顺序排队，全局递增序号（Redis INCR）保证公平性
> 3. **Lua 原子抢占**：用 Lua 脚本判断"你是否排在队列最前面"，原子操作防止多个等待者同时抢占导致超发信号量
> 4. **发布-订阅唤醒**：信号量释放时发 Redis 通知，等待者收到通知立即尝试，而不是傻等定时器（延迟更低）
> 5. **超时优雅拒绝**：等待超过配置时间（默认30秒）时，发送 SSE 拒绝消息并记录对话历史，用户体验不会"无响应"

```java
// ChatRateLimitAspect.java:57-76 - AOP 拦截 + 限流入口
@Around("@annotation(com.nageoffer.ai.ragent.rag.aop.ChatRateLimit)")
public Object limitStreamChat(ProceedingJoinPoint joinPoint) throws Throwable {
    Object[] args = joinPoint.getArgs();
    // 参数约定：args[0]=question, args[1]=conversationId, args[3]=SseEmitter
    if (args == null || args.length < 4 || !(args[3] instanceof SseEmitter emitter)) {
        return joinPoint.proceed();
    }
    String question = args[0] instanceof String q ? q : "";
    String actualConversationId = StrUtil.isBlank((String) args[1])
            ? IdUtil.getSnowflakeNextIdStr() : (String) args[1];
    args[1] = actualConversationId;

    Object target = joinPoint.getTarget();
    Method method = ((MethodSignature) joinPoint.getSignature()).getMethod();

    // 将真实业务逻辑封装成 Runnable 传入限流器（异步执行）
    chatQueueLimiter.enqueue(question, actualConversationId, emitter, () -> {
        invokeWithTrace(method, target, args, question, actualConversationId, emitter);
    });
    return null;  // SSE 是异步的，切面立即返回 null
}

// ChatQueueLimiter.java:111-143 - 排队入口
public void enqueue(String question, String conversationId, SseEmitter emitter, Runnable onAcquire) {
    if (!Boolean.TRUE.equals(rateLimitProperties.getGlobalEnabled())) {
        chatEntryExecutor.execute(onAcquire);  // 限流关闭，直接执行
        return;
    }

    String requestId = IdUtil.getSnowflakeNextIdStr();
    long seq = nextQueueSeq();  // Redis INCR 获取全局递增序号
    RScoredSortedSet<String> queue = redissonClient.getScoredSortedSet(QUEUE_KEY, StringCodec.INSTANCE);
    queue.add(seq, requestId);  // 入队：score=序号，value=requestId

    AtomicBoolean cancelled = new AtomicBoolean(false);
    AtomicReference<String> permitRef = new AtomicReference<>();

    // SSE 连接关闭时的释放逻辑（正常完成/超时/出错 三种情况）
    Runnable releaseOnce = () -> {
        cancelled.set(true);
        queue.remove(requestId);  // 出队
        String permitId = permitRef.getAndSet(null);
        if (permitId != null) {
            // 释放信号量，通知其他等待者
            redissonClient.getPermitExpirableSemaphore(SEMAPHORE_NAME).release(permitId);
            publishQueueNotify();  // 发布通知：有名额了
        }
    };
    emitter.onCompletion(releaseOnce);
    emitter.onTimeout(releaseOnce);
    emitter.onError(e -> releaseOnce.run());

    // 立即尝试获取信号量（有空闲名额时直接执行，无需等待）
    if (tryAcquireIfReady(queue, requestId, permitRef, cancelled, onAcquire)) {
        return;
    }

    // 没有空闲名额，注册轮询 + 订阅发布通知
    scheduleQueuePoll(queue, requestId, permitRef, cancelled, question, conversationId, ...);
}

// ChatQueueLimiter.java:239-250 - Redis PermitExpirableSemaphore
private String tryAcquirePermit() {
    RPermitExpirableSemaphore semaphore = redissonClient.getPermitExpirableSemaphore(SEMAPHORE_NAME);
    semaphore.trySetPermits(rateLimitProperties.getGlobalMaxConcurrent());  // 初始化最大并发数
    try {
        // tryAcquire(waitTime=0, leaseTime=90s)
        // leaseTime：信号量最多持有90秒，连接异常自动释放（防死锁）
        return semaphore.tryAcquire(0, rateLimitProperties.getGlobalLeaseSeconds(), TimeUnit.SECONDS);
    } catch (InterruptedException ex) {
        Thread.currentThread().interrupt();
        return null;
    }
}
```

**使用场景：** 防止高并发下大量 LLM 调用压垮服务，让多出的请求排队等待而不是直接报错。

**优化优点：**
- Redis 分布式信号量支持水平扩展，多实例统一限流
- 发布-订阅通知避免无谓的空轮询，资源占用低
- 租约过期防止因网络中断等意外导致的死锁

## 二、全链路可观测性 (@RagTraceNode)

### 整体流程：

```
RAGChatServiceImpl.streamChat() 被调用
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ ChatRateLimitAspect.invokeWithTrace()                                       │
│    ├── 生成 traceId（雪花ID）                                                │
│    ├── 写入 t_rag_trace_run（状态=RUNNING）                                  │
│    └── RagTraceContext.setTraceId(traceId)  // 写入当前线程 ThreadLocal      │
└─────────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ 方法链路执行（被 @RagTraceNode 标注的方法自动被 RagTraceAspect 拦截）        │
│    ├── rewriteWithSplit() → node_1 (REWRITE, depth=0, parentId=null)        │
│    ├── resolve()          → node_2 (INTENT,  depth=0, parentId=null)        │
│    │     └── classifyTargets() → node_2_1 (INTENT, depth=1, parentId=node_2)│
│    └── retrieveKnowledgeChannels() → node_3 (RETRIEVE, depth=0)             │
│              └── rerank() → node_3_1 (RERANK, depth=1, parentId=node_3)     │
└─────────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ ChatRateLimitAspect.invokeWithTrace() 完成                                  │
│    └── 更新 t_rag_trace_run（状态=SUCCESS/ERROR，总耗时）                    │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 核心类：

| 类名 | 职责 |
|------|------|
| `@RagTraceRoot` | 标注链路入口方法，每次请求创建一条 TraceRun 主记录 |
| `@RagTraceNode` | 标注链路中的每个关键环节，自动创建 TraceNode 节点记录 |
| `RagTraceContext` | ThreadLocal，存储当前线程的 traceId 和节点调用栈 |
| `RagTraceAspect` | AOP 切面，拦截注解方法，完成节点记录的开始/结束 |
| `RagTraceRecordService` | 异步写入追踪数据到 MySQL |
| `RagTraceRunDO` | 链路主记录（t_rag_trace_run）|
| `RagTraceNodeDO` | 节点记录（t_rag_trace_node）|

### 优化点：

| 优化项 | 说明 | 核心代码位置 |
|--------|------|--------------|
| 零侵入 | 业务代码无需写任何追踪逻辑，仅加注解 | `@RagTraceNode` 注解 |
| 父子节点树 | ThreadLocal 维护节点栈，实现嵌套调用的父子关系 | `RagTraceContext.pushNode()` / `popNode()` |
| 深度记录 | 记录节点调用深度，可视化为树形调用链 | `RagTraceContext.depth()` |
| 异步写入 | 追踪数据写入后台异步处理，不阻塞业务响应 | `RagTraceRecordService` |
| 开关控制 | `rag.trace.enabled=false` 全局关闭，切面直接透传 | `RagTraceProperties.isEnabled()` |
| 错误截断 | 异常信息超长时截断，防止大量错误堆栈占用存储 | `truncateError()` |

**代码示例：**

> **业务例子（伪代码）**
>
> ```
> 用户问："员工请假流程是什么？"
>
> 一次请求产生的追踪树（在 RagTraceQueryService 查询到的结构）：
>
>   [TraceRun] traceId="trace_001", status=SUCCESS, totalDuration=2340ms
>       │
>       ├── [Node] query-rewrite-and-split  (REWRITE,   depth=0, 320ms)  ← rewriteWithSplit()
>       │
>       ├── [Node] intent-resolve           (INTENT,    depth=0, 540ms)  ← resolve()
>       │       └── [Node] intent-classify  (INTENT,    depth=1, 490ms)  ← classifyTargets()
>       │
>       ├── [Node] retrieve-knowledge       (RETRIEVE,  depth=0, 890ms)  ← retrieveKnowledgeChannels()
>       │       ├── [Node] intent-directed  (SEARCH,    depth=1, 430ms)  ← IntentDirectedSearchChannel
>       │       └── [Node] rerank           (RERANK,    depth=1, 280ms)  ← RerankPostProcessor
>       │
>       └── [Node] stream-generate          (LLM,       depth=0, 590ms)  ← LLMService.streamChat()
>
>   从树中可以看出：
>   - 最慢环节：检索(890ms) → 可以排查 Rerank 模型或 TopK 配置
>   - LLM 生成(590ms) 只占总时长的25%，检索才是瓶颈
>   - 意图分类(490ms) 也较慢 → 可考虑意图树叶子节点数量是否过多
> ```
>
> **这段代码做什么？**
>
> 这是"注解式链路追踪"的核心实现，无侵入地记录每个环节的执行情况：
> 1. **入口创建链路**：`@ChatRateLimit` 切面在业务方法执行前创建 `RagTraceRunDO`，将 traceId 存入 ThreadLocal
> 2. **节点自动记录**：每个 `@RagTraceNode` 方法在执行前后，`RagTraceAspect` 自动创建 `RagTraceNodeDO`，记录耗时和状态
> 3. **父子关系维护**：通过 ThreadLocal 的节点栈（push/pop），进入方法时推栈，退出时弹栈，实现嵌套调用的父子关系
> 4. **异常安全捕获**：节点抛出异常时，切面记录 status=ERROR 和错误信息，**然后重新抛出**（不吞异常，业务仍然失败）

```java
// RagTraceAspect.java:117-173 - @RagTraceNode 拦截实现
@Around("@annotation(traceNode)")
public Object aroundNode(ProceedingJoinPoint joinPoint, RagTraceNode traceNode) throws Throwable {
    if (!traceProperties.isEnabled()) {
        return joinPoint.proceed();  // 开关关闭，直接透传，零开销
    }
    String traceId = RagTraceContext.getTraceId();
    if (StrUtil.isBlank(traceId)) {
        return joinPoint.proceed();  // 不在链路上下文中（如单元测试），直接透传
    }

    String nodeId = IdUtil.getSnowflakeNextIdStr();
    String parentNodeId = RagTraceContext.currentNodeId();  // 当前栈顶 = 父节点
    int depth = RagTraceContext.depth();                      // 栈深度 = 节点层级
    long startMillis = System.currentTimeMillis();

    Method method = ((MethodSignature) joinPoint.getSignature()).getMethod();

    // 创建节点记录（RUNNING 状态）
    traceRecordService.startNode(RagTraceNodeDO.builder()
            .traceId(traceId)
            .nodeId(nodeId)
            .parentNodeId(parentNodeId)  // 记录父子关系，构建树形结构
            .depth(depth)
            .nodeType(StrUtil.blankToDefault(traceNode.type(), "METHOD"))
            .nodeName(StrUtil.blankToDefault(traceNode.name(), method.getName()))
            .className(method.getDeclaringClass().getName())
            .methodName(method.getName())
            .status("RUNNING")
            .startTime(new Date())
            .build());

    RagTraceContext.pushNode(nodeId);  // 入栈，成为新的当前节点（子方法的父节点）
    try {
        Object result = joinPoint.proceed();
        traceRecordService.finishNode(traceId, nodeId, "SUCCESS", null,
                new Date(), System.currentTimeMillis() - startMillis);
        return result;
    } catch (Throwable ex) {
        traceRecordService.finishNode(traceId, nodeId, "ERROR",
                truncateError(ex), new Date(), System.currentTimeMillis() - startMillis);
        throw ex;  // 重新抛出，不吞异常，业务层仍然能感知到错误
    } finally {
        RagTraceContext.popNode();  // 出栈，恢复父节点为当前节点（无论成功还是失败）
    }
}
```

**使用场景：** 排查某次问答为何很慢、找出哪个环节报错、统计各环节平均耗时做性能优化。

**优化优点：**
- 加一个 `@RagTraceNode` 注解即可被追踪，对业务代码零侵入
- 父子节点树形结构直观展示"哪个子调用最慢"
- 开关控制保证高压力时可临时关闭追踪，不影响业务性能

## 三、MCP 工具生态

### 整体流程：

```
意图识别返回 kind=MCP 的节点
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ RetrievalEngine - MCP 分支                                                  │
│    ├── 从 MCPToolRegistry 找到对应的 MCPToolExecutor                         │
│    └── 调用 LLMMCPParameterExtractor.extractParameters()                    │
└─────────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ LLMMCPParameterExtractor.extractParameters()                                │
│    ├── 构建 Prompt：系统提示词 + 工具定义（参数名/类型/是否必填/描述）      │
│    ├── 调用 LLMService.chat()（低温度 0.1，保证稳定性）                     │
│    └── 解析 JSON 响应，只取工具声明的参数，填充默认值                        │
└─────────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ MCPToolExecutor.execute(MCPRequest)                                          │

    │
    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ ContextFormatter.formatMcpContext()                                          │
│    └── 将 MCPResponse.data 格式化为 Prompt 上下文字符串，注入给 LLM          │

### 核心类：

| 类名 | 职责 |
|------|------|
| `MCPTool` | 工具元数据定义（toolId、描述、参数定义列表 Map<String, ParameterDef>） |
| `MCPToolRegistry` | 工具注册表，Spring 启动时自动收集所有 MCPToolExecutor Bean |
| `MCPToolExecutor` | 工具执行器接口，一个实现类对应一类外部 API |
| `LLMMCPParameterExtractor` | 用 LLM 从用户问题中提取工具所需的参数（含默认值兜底） |
| `MCPRequest` / `MCPResponse` | 工具调用的请求/响应数据结构 |

### 优化点：

| 优化项 | 说明 | 核心代码位置 |
|--------|------|--------------|
| 工具自动注册 | Spring 容器启动时扫描所有 MCPToolExecutor Bean，自动注册 | `MCPToolRegistry` |
| LLM 参数提取 | 不需要固定规则，LLM 理解自然语言后提取参数 | `LLMMCPParameterExtractor.extractParameters()` |
| 自定义提示词 | 每个意图节点可配置 `paramPromptTemplate`，覆盖默认提取 Prompt | `extractParameters(..., customPromptTemplate)` |

| 全局异常兜底 | JSON 解析失败或 LLM 异常时，返回全默认值，工具调用仍能继续 | `buildDefaultParameters()` |

**代码示例：**

> **业务例子（伪代码）**
>
> ```
> 场景：意图识别命中"考勤查询"MCP工具节点
>
> MCPTool 定义（考勤查询工具）：
>   toolId: "attendance_query"
>   description: "查询员工的考勤打卡记录"
>   parameters:
>     - employeeId (string, 必填): 员工工号
>     - startDate  (string, 可选): 开始日期，格式yyyy-MM-dd，默认=本月1号
>     - endDate    (string, 可选): 结束日期，格式yyyy-MM-dd，默认=今天
>
> 用户问题："帮我查一下工号A001的3月考勤"
>
> LLMMCPParameterExtractor 处理：
>
>   第1步：构建 Prompt
>   [System]  "根据工具定义，从用户问题中提取参数，以JSON格式输出"
>   [User 1]  "工具定义如下：
>               工具ID: attendance_query
>               功能描述: 查询员工的考勤打卡记录
>               参数列表:
>                 - employeeId (类型: string, 必填): 员工工号
>                 - startDate  (类型: string, 可选): 开始日期 [默认值: 2024-03-01]
>                 - endDate    (类型: string, 可选): 结束日期 [默认值: 2024-03-31]"
>   [User 2]  "请根据以上工具定义，从下面的问题中提取参数：
>               帮我查一下工号A001的3月考勤"
>
>   第2步：LLM 返回
>   {"employeeId": "A001", "startDate": "2024-03-01", "endDate": "2024-03-31"}
>
>   第3步：安全解析（只取工具声明的参数）
>   tool.parameters 的 key 集合 = {employeeId, startDate, endDate}
>   extracted: {employeeId="A001", startDate="2024-03-01", endDate="2024-03-31"}
>
>   第4步：填充默认值
>   所有参数均已提取，fillDefaults() 无需操作
>
>   第5步：调用 MCPToolExecutor.execute()
>   MCPRequest{toolId="attendance_query", parameters={employeeId="A001", ...}}
>   → HTTP POST https://hr-system.corp.com/mcp/attendance/query
>   → MCPResponse{success=true, data={records:[{date:"03-01",status:"正常"},...], total:22}}
>
>   第6步：formatMcpContext()
>   将考勤数据格式化为文本 → 注入 LLM Prompt
>   → LLM 根据真实考勤数据生成回答："A001员工3月份共22条打卡记录，其中..."
> ```
>
> **这段代码做什么？**
>
> 这是"LLM 驱动的工具参数提取"核心实现，让工具调用支持自然语言输入：
> 1. **工具定义描述化**：将 MCPTool 的参数定义转成 LLM 能理解的文字描述（参数名、类型、是否必填、描述、枚举值、默认值）
> 2. **LLM 语义提取**：让 LLM 理解用户自然语言，从中抽取符合工具参数规范的值（如"3月"→ startDate/endDate）
> 3. **安全解析**：只提取工具定义中声明的参数，防止 LLM 幻觉出额外字段注入到工具调用
> 4. **默认值兜底**：可选参数 LLM 没有提取时，自动填充工具定义的默认值（如不指定日期，默认查本月）
> 5. **全局异常兜底**：JSON 解析失败或 LLM 异常时，返回全默认值，工具调用仍然能继续执行

// LLMMCPParameterExtractor.java:63-107 - 参数提取主逻辑
@Override
public Map<String, Object> extractParameters(String userQuestion, MCPTool tool, String customPromptTemplate) {
    if (tool == null || CollUtil.isEmpty(tool.getParameters())) {
        return Collections.emptyMap();
    }

    List<ChatMessage> messages = new ArrayList<>(3);
    // 优先使用意图节点配置的自定义 Prompt，否则用默认 Prompt
    String systemPrompt = StrUtil.isNotBlank(customPromptTemplate)
            ? customPromptTemplate
            : promptTemplateLoader.load(MCP_PARAMETER_EXTRACT_PROMPT_PATH);

    messages.add(ChatMessage.system(systemPrompt));
    messages.add(ChatMessage.user("工具定义如下：\n" + buildToolDefinition(tool)));  // 工具定义描述化
    messages.add(ChatMessage.user("请根据以上工具定义，从下面的问题中提取参数：\n" + userQuestion));

    String raw = null;
    try {
        ChatRequest request = ChatRequest.builder()
                .messages(messages)
                .temperature(0.1D)  // 低温度：保证参数提取的稳定性
                .topP(0.3D)
                .thinking(false)
                .build();
        raw = llmService.chat(request);

        Map<String, Object> extracted = parseJsonResponse(raw, tool);  // 安全解析
        fillDefaults(extracted, tool);  // 补充可选参数的默认值
        return extracted;
    } catch (JsonSyntaxException e) {
        log.warn("MCP 参数提取-JSON解析失败, toolId: {}, 响应: {}", tool.getToolId(), raw);
        return buildDefaultParameters(tool);  // 全默认值兜底，不让工具调用失败
    } catch (Exception e) {
        log.error("MCP 参数提取异常, toolId: {}", tool.getToolId(), e);
        return buildDefaultParameters(tool);  // 全默认值兜底
    }
}

// LLMMCPParameterExtractor.java:148-169 - 安全解析（只取工具声明的参数）
private Map<String, Object> parseJsonResponse(String raw, MCPTool tool) {
    String cleaned = LLMResponseCleaner.stripMarkdownCodeFence(raw);
    JsonObject obj = JsonParser.parseString(cleaned).getAsJsonObject();
    Map<String, Object> result = new HashMap<>();

    // 关键：遍历工具定义的参数 key，而非 JSON 的所有字段
    // 这样即使 LLM 返回了额外字段（幻觉），也不会被提取进来
    for (String paramName : tool.getParameters().keySet()) {
        if (obj.has(paramName) && !obj.get(paramName).isJsonNull()) {
            result.put(paramName, convertJsonElement(obj.get(paramName)));
        }
    }
    return result;
}

// LLMMCPParameterExtractor.java:118-142 - 工具定义文本化（供 LLM 理解）
private String buildToolDefinition(MCPTool tool) {
    StringBuilder sb = new StringBuilder();
    sb.append("工具ID: ").append(tool.getToolId()).append("\n");
    sb.append("功能描述: ").append(tool.getDescription()).append("\n");
    sb.append("参数列表:\n");

    for (Map.Entry<String, MCPTool.ParameterDef> entry : tool.getParameters().entrySet()) {
        MCPTool.ParameterDef def = entry.getValue();
        sb.append("  - ").append(entry.getKey());
        sb.append(" (类型: ").append(def.getType());
        sb.append(def.isRequired() ? ", 必填" : ", 可选");
        sb.append("): ").append(def.getDescription());
        if (def.getDefaultValue() != null) {
            sb.append(" [默认值: ").append(def.getDefaultValue()).append("]");
        }
        if (CollUtil.isNotEmpty(def.getEnumValues())) {
            sb.append(" [可选值: ").append(String.join(", ", def.getEnumValues())).append("]");
        }
        sb.append("\n");
    }
    return sb.toString();
}
```

**如何注册一个新的 MCP 工具：**

1. 实现 `MCPToolExecutor` 接口，声明为 Spring `@Component`
2. 在 `getToolDefinition()` 中定义工具的 `MCPTool`（toolId、参数规范）
3. 在 `execute(MCPRequest)` 中实现实际的 HTTP/数据库调用逻辑
4. 在意图树中添加一个 `kind=MCP` 的叶子节点，`mcpToolId` 指向第1步的 toolId
5. 服务启动时 `MCPToolRegistry` 自动发现并注册，无需修改其他代码

**使用场景：** 用户询问需要查询实时数据的问题（考勤、订单、库存、天气等），知识库中没有静态答案，但可以通过工具 API 实时获取。

**优化优点：**
- LLM 驱动参数提取，无需为每个工具写专用的正则/规则解析逻辑
- 自定义提示词让复杂参数提取更精准（特别是日期范围、枚举值等）
- 全兜底机制保证工具调用不会因参数提取失败而中断整个 RAG 流程
