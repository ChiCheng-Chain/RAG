### 2. RAG 核心流程

#### 2.1 Ragent 链路

一次用户提问，在 Ragent 里经过的完整链路如下：



### **🚀 核心角色与分工**

图中包含五个核心服务模块，它们像流水线一样协作完成任务：

1. **用户**：发起提问的终端。
2. **会话管理**：负责对话状态的维护，如初始化、加载历史记录等。
3. **意图理解**：对用户的问题进行预处理，包括重写、拆分和意图识别。
4. **检索引擎**：负责从知识库中查找相关信息。
5. **生成服务**：利用大模型生成最终回答。
6. **输出服务**：负责将结果推送给用户，并处理后续的持久化记录。

------

### **🔄 详细交互流程**

#### **第一阶段：会话初始化与问题预处理**

1. **会话初始化**：用户发起请求，**会话管理**模块首先进行初始化，加载相关的对话历史和当前上下文。
2. **问题传递**：会话管理将处理好的问题传递给**意图理解**模块。
3. **意图分析**：**意图理解**模块对问题进行重写、拆分，并识别用户的真实意图。
4. **兜底机制**：如果经过意图识别后，系统判断**置信度不足**（例如无法理解或需要引导），流程会直接返回给用户进行**意图引导**。如果**置信度充足**，则继续下一步。

#### **第二阶段：并行检索与数据整合**

1. **发起检索**：系统向**检索引擎**发送检索请求。

2. 并行搜索

   ：检索引擎采用

   并行处理

   策略，同时执行多种搜索任务：

   - **图表查询**：专门针对图表数据的搜索。
   - **全局向量检索**：基于向量相似度的全局搜索。
   - **MCP 工具调用**：调用外部工具或插件获取信息。
   - **去重与重排序**：对检索到的多源结果进行去重和相关性重排。

3. **返回结果**：处理后的检索结果返回给生成服务。

#### **第三阶段：答案生成与流式输出**

1. **Prompt 组装**：**生成服务**将系统设定、对话上下文、用户问题以及刚才检索到的相关内容组装成完整的 Prompt。

2. 模型路由与生成

   ：

   - 系统进入**循环**，根据路由策略调用合适的**检索模型**。
   - 设定**首包探测**超时时间为 60 秒。
   - **失败重试**：如果调用失败（例如 Token 不够用），系统会自动标记并切换到下一个模型重试。
   - **成功响应**：模型生成答案后，通过**流式输出**的方式实时推送给用户。

3. 并行后处理

   ：在流式输出的同时，系统会并行执行以下后台任务：

   - **SSE 流式推送**：确保前端实时接收数据。
   - **消息持久化**：将对话记录保存到数据库。
   - **标题生成**：为本次对话生成一个简洁的标题。
   - **Trace 记录**：记录全链路的追踪日志，用于监控和调试。

#### 2.3 模型路由与容错

生产环境不可能只依赖一个模型供应商，Ragent 的模型路由机制解决的就是这个问题：

这张图表的标题是“模型路由与容错机制”，它展示了一个系统如何选择模型并处理请求的流程，以及在出现问题时如何进行容错。

图表中有四个主要参与者（生命线）：

- ModelSelector（模型选择器）
- ChatClient（聊天客户端，括号内注明支持百炼、SiliconFlow、Ollama等）
- ProbeBufferingCallback（探测缓冲回调）
- HealthChecker（健康检查器）

整个流程被一个大的循环框（loop）包围，表示会“逐个尝试候选模型”。

**流程细节如下：**

1. **选择模型**：流程开始时，ModelSelector会“选择候选模型列表”，并按照“优先级排序”。
2. **发起请求**：接着，系统会“解析对应的 ChatClient”，然后向它“发起流式请求”，并“启动首包探测”。
3. **成功分支（绿色区域）**：
   - 如果在“60s 内收到首包”，则视为成功。
   - 系统会向 ProbeBufferingCallback 发送“成功”信号。
   - ProbeBufferingCallback 会“回放缓冲事件”。
   - 最后，系统“继续流式输出”，流程结束。
4. **失败分支（红色区域）**：
   - 如果出现“超时或异常”，则视为失败。
   - ChatClient 会向 ProbeBufferingCallback 报告“失败”。
   - ProbeBufferingCallback 会将请求“丢弃缓冲”。
   - 同时，系统会通知 HealthChecker “报告失败”。
   - HealthChecker 会进行“失败次数累计”。
   - 在 HealthChecker 内部有一个判断（alt）：如果“触发熔断阈值”，它会“标记不健康”并“触发熔断”。
   - 无论是否触发熔断，流程都会回到 ModelSelector，执行“尝试下一个候选”操作，进入下一次循环。

**总结：**
这个机制的核心逻辑是：首先按优先级尝试不同的模型。如果在规定时间内成功收到响应，就继续处理；如果失败（超时或出错），就记录失败、检查是否需要熔断该模型，并自动切换到下一个候选模型进行重试。



关键设计：首包探测阶段会缓冲所有事件，确保模型切换时用户端不会收到半截的脏数据。

### 3. 文档入库流水线

文档从上传到可检索，经过一条基于节点编排的 Pipeline：



### **文档处理流水线**

这是一个用于将原始文档转化为可被检索和分析的结构化数据的完整流程。它由六个顺序执行的节点组成：

1. **FetcherNode（抓取节点）**
   - **功能**：负责获取原始内容。
   - **输入来源**：支持从多种渠道抓取，包括URL链接、本地文件（File）或云存储服务（如S3）。
2. **ParserNode（解析节点）**
   - **功能**：对获取到的原始内容进行文本解析。
   - **技术实现**：图中特别标注使用了Apache Tika技术，这是一种常用的用于从各种文件格式中提取元数据和文本内容的工具。
3. **EnhancerNode（增强节点）**
   - **功能**：对解析后的文本内容进行增强处理。
   - **处理方式**：主要包括“清洗”和“格式化”，目的是去除噪声、统一格式，提升后续处理的质量。
4. **ChunkerNode（分块节点）**
   - **功能**：将处理好的文本进行“分块”。
   - **处理方式**：根据“语义切分”逻辑，将长文本分割成更小、更易于管理的片段（chunks），这在处理大语言模型时尤其重要。
5. **EnricherNode（丰富节点）**
   - **功能**：对分块后的文本片段进行进一步的丰富和增强。
   - **处理方式**：包括“片段增强”和“标签化”，例如为每个文本块添加元数据、关键词或分类标签，同时进行“元数据注入”，使其包含更多上下文信息。
6. **IndexerNode（索引节点）**
   - **功能**：将最终处理好的数据进行索引并存储。
   - **处理方式**：执行“向量化 + 写入”操作。向量化是将文本转换为数值向量，便于进行语义相似度搜索；写入则是将这些向量数据存入向量数据库（图中示例为Milvus）。

------

这个流水线设计确保了从原始文档到可检索知识库的高效、自动化处理。每个节点各司其职，共同完成了数据的提取、清洗、结构化和索引化。

每个节点的配置存储在数据库中，支持条件执行和输出链式传递。每个任务和节点都有独立的执行日志，出了问题能精确定位到哪一步。

Ragent 不是为了用设计模式而用，每个模式都对应一个具体的工程问题：

#### 4.3 注册表模式

**问题：** MCP 工具数量不定，每次新增工具都要修改中心调度逻辑，成本高。

**接口**（[MCPToolRegistry.java](bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/MCPToolRegistry.java)）：

```java
public interface MCPToolRegistry {
    void register(MCPToolExecutor executor);
    Optional<MCPToolExecutor> getExecutor(String toolId);
    List<MCPTool> listAllTools();
}
```

**自动发现实现**（[DefaultMCPToolRegistry.java](bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/DefaultMCPToolRegistry.java)）：

```java
@Component
public class DefaultMCPToolRegistry implements MCPToolRegistry {
    // Spring 自动注入所有 MCPToolExecutor Bean
    private final List<MCPToolExecutor> autoDiscoveredExecutors;

    @PostConstruct
    public void init() {
        for (MCPToolExecutor executor : autoDiscoveredExecutors) {
            register(executor); // 启动时统一注册，新增工具只需声明 @Component
        }
    }
}
```

新增一个 MCP 工具只需实现 `MCPToolExecutor` 接口并加 `@Component`，注册表自动发现，无需修改任何配置。`IntentNodeRegistry`（[IntentNodeRegistry.java](bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/intent/IntentNodeRegistry.java)）对意图节点同理。

#### 4.4 模板方法

**问题：** 每个入库节点的执行流程结构相同（条件检查 → 计时 → 执行 → 记录日志 → 错误处理），但核心逻辑各异。

**固定骨架**（[IngestionEngine.java](bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/engine/IngestionEngine.java)）：

```java
private NodeResult executeNode(IngestionContext context, NodeConfig nodeConfig) {
    // 固定步骤1：条件检查
    if (nodeConfig.getCondition() != null && !conditionEvaluator.evaluate(...)) {
        return NodeResult.skip("条件未满足");
    }
    // 固定步骤2：计时 + 执行（变化点）
    long start = System.currentTimeMillis();
    try {
        NodeResult result = node.execute(context, nodeConfig); // ← 子类各自实现
        // 固定步骤3：记录日志
        context.getLogs().add(NodeLog.builder()...durationMs(duration)...build());
        return result;
    } catch (Exception e) {
        // 固定步骤4：统一错误处理
        return NodeResult.fail(e);
    }
}
```

**各节点只关注核心逻辑**（以 [FetcherNode.java](bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/node/FetcherNode.java) 为例）：

```java
@Component

        // 只写取文档的逻辑，计时/日志/错误处理由引擎统一完成
        DocumentFetcher fetcher = fetchers.get(source.getType());
        FetchResult result = fetcher.fetch(source);
        context.setRawBytes(result.content());
        return NodeResult.ok("已获取 " + result.content().length + " 字节");
    }
}
```

#### 4.5 装饰器模式

**问题：** 模型流式输出时，若第一包超时或出错就切换模型，必须保证已输出内容不会污染客户端。

**实现**（[RoutingLLMService.java](infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/RoutingLLMService.java) 内部类 `ProbeBufferingCallback`）：

```java
// 装饰器：包裹真正的 downstream callback，在探测阶段拦截所有事件
private static final class ProbeBufferingCallback implements StreamCallback {
    private final StreamCallback downstream; // 被装饰对象
    private final List<BufferedEvent> bufferedEvents = new ArrayList<>();
    private volatile boolean committed;

    @Override
    public void onContent(String content) {
        awaiter.markContent();
        bufferOrDispatch(BufferedEvent.content(content)); // 未 commit 前只缓存
    }

    /** 首包探测成功后提交：原子切换 + 按序回放缓存 */
    private void commit() {
        synchronized (lock) { committed = true; }
        for (BufferedEvent event : snapshot) {
            dispatch(event); // 转发给真正的 downstream
        }
    }
}
```

使用侧：

```java
// RoutingLLMService.streamChat()
ProbeBufferingCallback wrapper = new ProbeBufferingCallback(callback, awaiter);
handle = client.streamChat(request, wrapper, target); // 传入包装后的 callback
if (result.isSuccess()) {
    wrapper.commit(); // 探测成功，释放缓冲
}
```

#### 4.7 观察者模式

#### 4.8 AOP

**问题：** 链路追踪和限流逻辑散布在所有业务方法中，既污染代码又难以统一开关。

**@RagTraceNode 注解**（[RagTraceNode.java](framework/src/main/java/com/nageoffer/ai/ragent/framework/trace/RagTraceNode.java)）：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RagTraceNode {
    String name() default "";
    String type() default "METHOD";
}
```

**切面实现**（[RagTraceAspect.java](bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/aop/RagTraceAspect.java)）：

```java
@Around("@annotation(traceNode)")
public Object aroundNode(ProceedingJoinPoint joinPoint, RagTraceNode traceNode) {
    String nodeId = IdUtil.getSnowflakeNextIdStr();
    traceRecordService.startNode(...); // 自动记录节点开始
    RagTraceContext.pushNode(nodeId);
    try {
        return joinPoint.proceed();    // 执行业务方法
    } finally {
        RagTraceContext.popNode();      // 自动维护调用栈深度
        traceRecordService.finishNode(...);
    }
}
```

使用时只需一行注解，追踪逻辑零侵入：

```java
// MultiChannelRetrievalEngine.java
@RagTraceNode(name = "multi-channel-retrieval", type = "RETRIEVE_CHANNEL")
public List<RetrievedChunk> retrieveKnowledgeChannels(...) { ... }

// RoutingLLMService.java
@RagTraceNode(name = "llm-stream-routing", type = "LLM_ROUTING")
public StreamCancellationHandle streamChat(...) { ... }
```

`@ChatRateLimit`（[ChatRateLimitAspect.java](bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/aop/ChatRateLimitAspect.java)）同理，切面拦截标注了 `@ChatRateLimit` 的 SSE 入口，将请求放入队列限流，业务代码无感知。
